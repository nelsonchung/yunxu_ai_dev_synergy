# Case Plan Verification

此文件用於驗證「專案開發」設計是否符合 `docs/case_plan.md` 的流程規劃。

## 1. 專案列表頁

路徑：`/projects`

- 應顯示專案卡片列表，包含：專案名稱、客戶、階段、狀態、更新時間、託管狀態
- 右上角應有「新增需求」按鈕，導向 `/projects/new`
- 可見搜尋欄位與狀態/角色篩選（UI 即可）
- 至少三筆示範資料與不同狀態

## 2. 新增需求頁

路徑：`/projects/new`

- 有完整表單欄位：專案名稱、客戶名稱、預算、時程、需求描述
- 有「✨ AI 幫我完善需求」按鈕
- 點擊按鈕可展開 AI 草案區塊
- 頁面含流程預覽（需求釐清 → 媒合 → 規劃 → 交付）

## 3. 專案詳情頁

路徑：`/projects/:id`（例如 `/projects/yx-241`）

- 顯示專案摘要、標籤、當前狀態與託管狀態
- 有「狀態時間軸」且可辨識當前節點
- 有「專案文件」區塊，顯示文件類型與狀態
- 有「狀態紀錄」區塊
- 顯示「條件檢查」清單（Guard Conditions）

## 4. 狀態轉換與退回

- 「下一步可執行狀態」區塊應顯示下一個狀態
- 若權限或 Guard 不符合，狀態應呈現 disabled 提示文案
- 退回節點需標示「退回節點」

## 5. 託管邏輯檢查

以 `/projects/yx-241` 為例：
- 託管狀態顯示為「等待託管」
- 「資金託管確認 → 專案啟動」應顯示 guard 提示（款項未到位）
- 若後續資料改成 `paymentStatus = ESCROWED`，此 guard 應顯示可執行

## 6. 角色切換示意

在專案詳情頁：
- 切換角色（客戶/開發團隊/管理者）應影響「下一步可執行狀態」的顯示
- 管理者角色應可視為全開放

## 7. 導覽一致性

- Navbar 與 Footer 有「專案管理」入口，可前往 `/projects`
- 專案詳情頁有「返回專案列表」連結

## 8. 視覺風格一致性

- 新頁面需符合品牌視覺（字體、色系、圓角、卡片陰影）
- 背景需有漸層/紋理，不可全白
