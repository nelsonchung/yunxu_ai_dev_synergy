# 系統驗證文件（System Verification）

## 1. 文件目的
本文件依據 `docs/requirement/customer_requirement.md`、`docs/requirement/system_architecture.md` 與 `docs/requirement/software_design.md`，定義「鋆旭 AI-Dev 軟體系統媒合平台」的驗證策略、測試範圍、測試項目與驗收標準。

## 2. 驗證範圍
- 需求對齊：需求提交、文件化、簽核流程。
- 媒合評估：估工與團隊媒合流程。
- 協作開發：文件版本控管、任務追蹤。
- 品質交付：AI Code Review、測試文件、品質報告。
- 權限與稽核：RBAC、審計記錄、資料隔離。

## 3. 驗證策略
- 以流程為主：對應「需求對齊 -> 媒合評估 -> 協作開發 -> 品質交付」。
- 以角色為主：客戶、合作團隊、管理者使用路徑皆需驗證。
- 以可追溯為主：文件版本、簽核、AI 產出必須被記錄。
- 以可驗證為主：測試輸出文件與品質報告可被追蹤與比對。

## 4. 測試類型
- 功能測試：各模組核心功能與流程驗證。
- 整合測試：跨服務與資料流完整性。
- 權限測試：角色與資料隔離。
- AI 協作測試：AI 任務觸發、結果回寫、人工覆核。
- 穩定度測試：主要流程與文件操作的穩定性。
- 可用性檢核：主要頁面操作流程清楚可理解。

## 5. 驗證環境與資料
- 環境：測試環境（與正式環境設定一致）。
- 使用者：客戶帳號、合作團隊帳號、管理者帳號。
- 測試資料：需求案例、文件樣本、測試報告樣本。

## 6. 需求對照與驗證矩陣（概要）
| 需求項目 | 驗證方式 | 驗收標準 |
| --- | --- | --- |
| 需求文件化 | 功能測試 | 可建立需求並產出文件版本 |
| 需求簽核 | 功能/權限 | 只有有權角色可簽核 |
| 媒合評估 | 功能/整合 | 評估結果可產出並可追蹤 |
| 文件版本控管 | 功能/稽核 | 新版本建立後舊版可追溯 |
| 任務追蹤 | 功能 | 任務狀態可更新且可視化 |
| AI Code Review | AI/整合 | 可觸發任務並產出報告 |
| 測試文件生成 | AI/整合 | 可生成測試文件並保存 |
| 稽核紀錄 | 功能/安全 | 所有關鍵異動可查詢 |

## 7. 核心流程驗證
### 7.1 需求對齊
- 客戶建立需求並提交。
- 需求文件產出、版本可檢視。
- 管理者或指定角色完成簽核。

### 7.2 媒合評估
- 需求文件輸入媒合流程。
- 產出媒合結果、估工與評分。

### 7.3 協作開發
- 產生系統開發文件與軟體設計文件。
- 任務拆解、狀態更新、里程碑追蹤。

### 7.4 品質交付
- 產生測試文件。
- 觸發 AI Code Review，產出品質報告。
- 交付文件彙整並完成驗收。

## 8. 代表性測試案例
### 8.1 需求提交與文件化
- 前置條件：客戶已登入。
- 步驟：提交需求 -> 系統產出需求文件。
- 預期結果：需求文件可下載，版本為 v1。

### 8.2 需求簽核權限
- 前置條件：需求文件待簽核。
- 步驟：一般客戶嘗試簽核 -> 管理者簽核。
- 預期結果：一般客戶被拒絕，管理者簽核成功。

### 8.3 媒合評估
- 前置條件：需求文件已簽核。
- 步驟：啟動媒合評估。
- 預期結果：產生媒合結果與估工報告。

### 8.4 文件版本控管
- 前置條件：已有文件 v1。
- 步驟：新增文件版本。
- 預期結果：產生 v2，v1 標記為 archived。

### 8.5 AI Code Review
- 前置條件：專案已建立並有程式碼來源。
- 步驟：觸發 AI Code Review。
- 預期結果：產生 AI 審查報告與狀態完成。

### 8.6 測試文件生成
- 前置條件：專案建立完成。
- 步驟：觸發測試文件生成。
- 預期結果：測試文件可下載並可版本追蹤。

## 9. 驗收標準
- 主要流程均可完成且狀態正確。
- 文件版本、簽核與稽核紀錄可追溯。
- AI 任務可觸發、完成並保留結果。
- 角色權限控管符合規格。

## 10. 測試交付物
- 測試案例清單。
- 測試報告與結果。
- AI 任務驗證報告。
- 稽核紀錄與抽查結果。

## 11. 風險與注意事項
- AI 結果需人工覆核，避免誤用。
- 需求資訊不足會影響評估與測試準確度。
- 大型專案需注意負載與延遲風險。
